## 摘要
我们介绍了 Agent^2 RL-Bench，这是一个用于评估智能体化强化学习（RL）后训练的基准测试，旨在考察大语言模型（LLM）智能体能否自主设计、实现并运行完整的 RL 流水线以提升基础模型。这一能力至关重要，因为 RL 后训练正日益成为驱动模型对齐和专业化的核心，然而现有的基准测试大多保持静态：仅靠有监督微调（SFT）就能取得显著效果，导致交互式 RL 工程化能力未得到测试。Agent^2 RL-Bench 通过三个层级的六个任务解决了这一问题——从静态的基于规则的训练到包含轨迹采集的闭环在线 RL，每个层级都增加了前一层级所不具备的结构性要求。该基准测试提供了隔离的工作空间和评分 API、记录每次提交和代码修订的运行时监测工具，以及生成结构化运行报告的自动化事后分析，从而实现了对智能体驱动的后训练行为的首次自动化诊断。通过对涵盖 5 个智能体系统和 6 个驱动 LLM 的多个智能体栈进行测试，我们发现智能体在交互式任务中取得了显著收益——在 ALFWorld 上，一个纯 RL 智能体通过 SFT 预热和带有在线采样的 GRPO，性能从 5.97 提升至 93.28；但在其他任务上进展甚微（如 DeepSearchQA 仅提升 2.75，处于评估噪声范围内）。此外，驱动模型的选择对交互式任务影响巨大——在相同的框架下，更换驱动模型可使交互式改进从接近零提升至 78 个百分点。更广泛地看，该基准测试揭示了在固定预算下，有监督流水线在智能体驱动的后训练中占据主导地位，而在线 RL 仅在 ALFWorld 上作为最终的最佳路径取得了成功。代码可在 https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench 获取。

## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. Code is available at https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench.

*   **核心问题**：LLM 智能体能否在有限的工程预算下，自主完成从任务理解、策略选择、代码实现、环境交互（Rollout）、奖励诊断到模型迭代的完整 RL 后训练闭环？
*   **研究动机**：填补静态基准测试的空白，揭示智能体在处理需要状态交互、轨迹采集和在线数据生成的“智能体化 RL”任务时的真实水平。

#### 2. 方法论
*   **核心思想**：构建一个具有“训练结构覆盖度”的基准测试，将任务按复杂度分为三个层级，强制智能体面对从静态验证到闭环在线 RL 的转变。
*   **关键技术细节**：
    *   **三层级结构**：
        *   **L1（静态规则）**：GSM8K, HumanEval（确定性验证）。
        *   **L2（静态评判）**：AlpacaEval 2.0（非确定性 LLM 评判）。
        *   **L3（交互采样）**：ALFWorld, WebShop, DeepSearchQA（需要环境交互、多步轨迹采集和长程奖励推理）。

        *   **隔离工作区**：为每个智能体提供独立环境，包含基础模型和训练数据。
        *   **评分 API**：支持多次提交，智能体可根据反馈进行诊断和修复。
        *   **运行时监测**：记录所有代码修订和模型工件。
        *   **自动化事后分析**：自动生成结构化运行报告，用于诊断智能体的搜索行为和失败模式。

#### 3. 实验设计
*   **数据集/场景**：涵盖数学推理（GSM8K）、代码生成（HumanEval）、指令遵循（AlpacaEval）、具身智能交互（ALFWorld）、电商搜索交互（WebShop）和深度搜索问答（DeepSearchQA）。

*   **驱动模型（Drivers）**：使用了 GPT-5.4、GPT-5.2、GPT-4o、Claude Opus 4.6、Claude Sonnet 4.5 和 Gemini 2.5-flash。
*   **对比方法**：通过设置不同的操作模式（SFT-only, RL-only, Free, Multi-task）进行消融实验。

*   **算力资源**：实验中明确提到了使用 **NVIDIA H200 GPU**（附录 A.19/A.22）。

#### 6. 主要结论与发现
*   **交互式增益显著但非普遍**：在 ALFWorld 上，智能体能将模型性能从 5.97 提升至 93.28；但在 DeepSearchQA 上，所有智能体均难以突破性能天花板。
*   **驱动模型是关键变量**：在相同框架下，更换驱动模型（如从 Sonnet 换成 Opus）可使 ALFWorld 的改进幅度从 6.7% 飙升至 88.8%。

*   **对齐差距的闭合依赖全栈**：从非对齐的基础模型出发，只有最强的“框架+驱动”组合能达到或超过指令微调（Instruct）版本的水平。

#### 7. 优点
*   **结构化诊断**：不仅关注最终得分，还通过运行时监测和自动化报告分析智能体的“心路历程”。
*   **任务分级科学**：通过 L1-L3 的分级，清晰地界定了静态自动化与动态 RL 工程化之间的能力边界。
*   **真实工程模拟**：允许重复提交和错误修复，贴合现实中算法工程师的工作流程。

#### 8. 不足与局限
*   **评估噪声**：DeepSearchQA 和 AlpacaEval 存在较高的评估方差（±3pp），可能掩盖微小的算法改进。
*   **实验覆盖限制**：受限于算力和成本，8B 实验缺乏多次重复的置信区间分析。
*   **策略偏好偏差**：智能体在面对复杂 RL 任务时表现出明显的“退缩”行为，倾向于回退到 SFT，这可能限制了对纯 RL 工程能力的深度挖掘。
*   **API 依赖**：基准测试高度依赖闭源模型的 API 稳定性（如 Azure 内容过滤曾干扰 AlpacaEval 的评判）。
