## 摘要
我们推出了 Agent^2 RL-Bench，这是一个用于评估智能体强化学习（RL）后训练的基准测试，旨在考察大语言模型（LLM）智能体能否自主设计、实现并运行完整的强化学习流水线，以提升基础模型的能力。这一能力至关重要，因为强化学习后训练正日益成为模型对齐和专业化的驱动力，然而现有的基准测试大多是静态的：仅靠监督微调（SFT）就能取得显著成效，导致交互式强化学习工程化能力未得到测试。Agent^2 RL-Bench 通过三个层级的六个任务解决了这一问题——从静态的基于规则的训练到包含轨迹采集的闭环在线强化学习——每个层级都增加了前一层级所不具备的结构性要求。该基准测试提供了带有评分 API 的隔离工作区、记录每次提交和代码修订的运行时检测工具，以及生成结构化运行报告的自动化事后分析，从而实现了对智能体驱动的后训练行为的首次自动化诊断。在涵盖 5 个智能体系统和 6 个驱动 LLM 的多个智能体堆栈中，我们发现智能体取得了显著的交互式增益——在 ALFWorld 上，一个纯强化学习智能体通过 SFT 预热和带有在线 Rollout 的 GRPO，性能从 5.97 提升至 93.28——但在其他任务上进展甚微（如 DeepSearchQA 仅提升 2.75，处于评估噪声范围内）；此外，驱动模型的选择对交互式任务影响巨大——在相同的框架下，更换驱动模型可使交互式改进从接近零变为提升 78 个百分点。更广泛地说，该基准测试揭示了在固定预算下，监督流水线在智能体驱动的后训练中占据主导地位，而在线强化学习仅在 ALFWorld 上成为了最终的最佳路径。代码已在 https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench 开源。

## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. Code is available at https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench.

### 1. 核心问题与研究动机
*   **核心问题**：LLM 智能体是否能够自主地设计、实现、调试并运行一整套强化学习（RL）流水线，从而提升另一个基础模型的能力？
*   **研究动机**：
    *   **RL 后训练的重要性**：RL（如 PPO、GRPO）已成为提升模型推理能力和对齐的关键技术，但其工程实现极其复杂，涉及奖励函数设计、算法稳定性控制和在线数据采集。
    *   **现有基准的局限性**：现有的智能体基准（如 SWE-bench 或 MLE-bench）大多偏向静态任务或简单的监督微调（SFT）。在这些基准下，智能体往往只需写出 SFT 代码即可获得高分，无法真正测试其处理“在线交互”和“轨迹级奖励”的能力。
    *   **自动化需求**：如果智能体能胜任此任务，将极大地加速模型迭代效率。

### 2. 核心方法论
论文提出了 **Agent^2 RL-Bench**，其核心设计思想是**训练结构覆盖（Training Structure Coverage）**：
*   **三层级任务结构**：
    *   **L1（静态规则级）**：如 GSM8K、HumanEval。测试基础的数据构建和确定性验证下的 SFT/RL。
    *   **L2（静态评判级）**：如 AlpacaEval 2.0。引入非确定性的 LLM 评判信号，测试智能体对模糊奖励的优化能力。
    *   **L3（交互式 Rollout 级）**：核心挑战。包括 ALFWorld、WebShop 和 DeepSearchQA。要求智能体实现环境交互、多步轨迹采集和闭环在线 RL。
*   **系统架构**：
    *   **隔离工作区**：为智能体提供只读的基础模型和训练数据，以及可写的工作目录。
    *   **评分 API**：智能体可多次提交模型进行评估，获取反馈并迭代。
    *   **运行时检测与诊断**：记录智能体的每一次代码修改、提交记录和模型工件，并自动生成结构化的事后分析报告（Run Reports），用于诊断智能体的失败原因（如 Rollout 失败、奖励设计错误等）。

### 3. 实验设计
*   **数据集/场景**：涵盖数学推理（GSM8K）、代码生成（HumanEval）、指令遵循（AlpacaEval）以及三个复杂的交互式环境（ALFWorld 具身智能、WebShop 购物决策、DeepSearchQA 深度搜索）。
*   **对比方法（智能体堆栈）**：
    *   **Scaffolds（框架）**：OpenHands、OpenCode、Claude Code、Codex CLI、Gemini CLI。
    *   **Driver LLMs（驱动模型）**：包括 GPT-5.4、GPT-5.2、Claude Opus 4.6、Claude Sonnet 4.5、Gemini 2.5-flash 等（注：论文标注日期为 2026 年，模型名称反映了其前瞻性设定）。
*   **基准线**：以 Qwen2.5-7B-Instruct 和 Qwen3-8B-Base 作为被训练的基础模型（Base Model）。

### 4. 资源与算力
*   **工程预算**：每个任务为智能体设定了严格的 **12 小时**（或部分实验 24 小时）固定工程预算。
*   **硬件支持**：附录提到使用了 **H200 GPU** 集群进行训练。
*   **Token 消耗**：详细记录了不同智能体的 Token 消耗量，例如 Claude Code 在 DeepSearchQA 任务中消耗了约 75M 个 Token，成本约 210 美元。

### 5. 实验数量与充分性
*   **实验规模**：进行了跨 5 个系统、6 个驱动模型、6 个任务的多维度对比。
*   **消融实验**：针对 Claude Code 进行了“操作模式”消融（SFT-only, RL-only, Free, Multi-task），探讨了不同训练范式对结果的影响。
*   **客观性**：
    *   引入了 **pass@k** 分析，评估智能体成功的偶然性。
    *   对 DeepSearchQA 进行了 20 次重复实验以评估评估噪声（±3pp）。
    *   通过隔离测试集防止数据泄露。
*   **充分性**：实验不仅关注最终得分，还深入分析了智能体的代码演变轨迹和失败分类，具有很高的诊断价值。

### 6. 主要结论与发现
1.  **交互式增益显著但非均一**：在 ALFWorld 上，智能体能将模型性能从 5.97 提升至 93.28（通过 GRPO 在线 RL）；但在 DeepSearchQA 上几乎没有实质性进展。
2.  **驱动模型是关键**：在相同框架下，更换驱动模型（如从 GPT-4o 换成 GPT-5.2）能让 ALFWorld 的改进幅度从 0 飙升至 78 个百分点。
3.  **SFT 仍是“避风港”**：在固定预算下，智能体更倾向于构建监督学习流水线。在线 RL 仅在 ALFWorld 这一特定任务中成为最佳路径。
4.  **对齐差距的闭合依赖全堆栈**：对于未对齐的基础模型（8B-Base），只有最强的智能体堆栈能将其提升至 Instruct 级别，弱智能体甚至会导致模型性能倒退。

### 7. 优点
*   **结构化诊断**：不仅给出一个分数，还通过自动化报告揭示了智能体“如何”思考和失败。
*   **填补空白**：首次系统性地评估了智能体在“在线交互式 RL”这一高难度工程领域的表现。
*   **模式控制**：通过强制智能体进入特定训练模式（如 RL-only），揭示了不同算法范式的潜力边界。

### 8. 不足与局限
*   **评估噪声**：部分任务（如 DeepSearchQA）的评估信号存在较大波动，可能干扰智能体的迭代决策。
*   **计算成本高昂**：运行此类基准测试需要大量的 GPU 资源和 API 调用成本，限制了其在大规模快速验证中的普及。
*   **任务覆盖面**：虽然涵盖了三个层级，但任务总量（6 个）相对较少，可能无法完全代表现实中千变万化的后训练需求。
*   **模型名称前瞻性**：论文中使用的部分模型名称（如 GPT-5.4）在当前现实中尚未发布，可能属于模拟实验或特定内部版本的代号。
