## 摘要
我们推出了 Agent^2 RL-Bench，这是一个用于评估智能体强化学习（agentic RL）后训练的基准测试，旨在考察大语言模型（LLM）智能体能否自主设计、实现并运行完整的强化学习流水线，从而提升基础模型的能力。这一能力至关重要，因为强化学习后训练正日益成为模型对齐和专业化的驱动力，然而现有的基准测试大多仍是静态的：仅靠监督微调（SFT）就能取得显著成果，导致交互式强化学习工程化能力未得到测试。Agent^2 RL-Bench 通过三个等级的六个任务解决了这一问题——从静态的基于规则的训练到带有轨迹采集的闭环在线强化学习——每个等级都增加了前一等级所不具备的结构性要求。该基准测试提供了带有评分 API 的隔离工作区、记录每次提交和代码修订的运行时检测工具，以及生成结构化运行报告的自动化事后分析，从而实现了对智能体驱动的后训练行为的首次自动化诊断。在涵盖 5 个智能体系统和 6 个驱动 LLM 的多个智能体堆栈中，我们发现智能体取得了显著的交互式增益——在 ALFWorld 上，一个纯强化学习智能体通过 SFT 预热和带有在线展开（online rollouts）的 GRPO，其性能从 5.97 提升至 93.28——但在其他任务上仅取得微小进展（如 DeepSearchQA 仅提升 2.75，处于评估噪声范围内）；此外，驱动模型的选择对交互式任务有巨大影响——在相同的框架下，更换驱动模型可使交互式改进从接近零变为提升 78 个百分点。更广泛地说，该基准测试揭示了在固定预算下，监督流水线在智能体驱动的后训练中占据主导地位，而在线强化学习仅在 ALFWorld 上作为最终的最佳路径取得了成功。

## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. ### 2. 方法论：Agent^2 RL-Bench 框架
*   **核心思想**：通过“训练结构覆盖（Training Structure Coverage）”原则，设计一个从简单到复杂的递进式评估体系。
*   **三级任务结构**：
    *   **L1 静态规则级**：使用确定性校验器（如 GSM8K, HumanEval），测试基础的数据构建和监督训练能力。
    *   **L2 静态评判级**：引入非确定性信号（如 AlpacaEval 2.0），要求智能体针对 LLM 评判员的偏好进行优化。
    *   **L3 交互展开级**：核心挑战。要求智能体与状态环境交互、采集多步轨迹并处理稀疏/稠密奖励（如 ALFWorld, WebShop, DeepSearchQA）。
*   **关键技术细节**：
    *   **隔离工作区**：为智能体提供独立环境，包含基础模型、训练数据和评分 API。
    *   **运行时检测（Instrumentation）**：实时记录智能体的每一次代码修改、模型提交和运行日志。
    *   **自动化事后分析**：自动生成结构化运行报告，诊断智能体是死于代码错误、奖励设计失败还是训练不稳定。

### 3. 实验设计
*   **数据集/场景**：
    *   **L1**：GSM8K（数学）、HumanEval（代码）。
    *   **L2**：AlpacaEval 2.0（指令遵循）。
    *   **L3**：ALFWorld（具身交互）、WebShop（在线购物交互）、DeepSearchQA（工具增强搜索）。
*   **对比方法（智能体堆栈）**：
    *   **5 种智能体框架（Scaffolds）**：OpenHands, OpenCode, Claude Code, Codex CLI, Gemini CLI。
    *   **6 种驱动模型（Drivers）**：包括 Claude 3.5/4.6 系列、GPT-4o/5.x 系列、Gemini 2.5-flash。
*   **评估指标**：主要指标是改进值 $\Delta$（提交模型得分 - 基础模型得分），以及成功率和过程指标（提交次数、有效提交率等）。

### 4. 资源与算力
*   **时间预算**：每个任务为智能体分配固定的 **12 小时**（部分实验为 24 小时）工程预算。
*   **硬件资源**：附录提到实验使用了 **NVIDIA H200 GPU**（单机双卡或多卡环境），用于支持智能体频繁启动的训练作业。
*   **算力消耗**：论文详细记录了 Token 消耗量，例如 Claude Code 在 24 小时内处理了约 2.9 亿个 Token，估计成本约 796 美元。

### 5. 实验数量与充分性
*   **实验规模**：涵盖了 5 个系统、6 个驱动模型在 6 个任务上的表现，并针对 Qwen3-8B-Base 进行了受控变量研究。
*   **消融实验**：对 Claude Code 的“操作模式”（纯 SFT、纯 RL、自由模式、多任务模式）进行了深入消融。
*   **充分性评价**：实验设计较为全面，不仅关注最终得分，还通过“Pass@k”分析和轨迹审计来探讨偶然性。但作者也承认，部分 8B 模型的受控实验为单次运行，且 DeepSearchQA 存在较高的评估噪声（±3pp）。

### 6. 主要结论与发现
*   **交互式增益显著但极不均衡**：在 ALFWorld 上，智能体能将模型得分从 5.97 提升至 93.28；但在 DeepSearchQA 上几乎没有实质性进展。
*   **驱动模型是决定性因素**：在相同框架下，更换驱动模型（如从 GPT-4o 换成更强版本）可使 ALFWorld 的改进幅度从 0.74pp 飙升至 78.35pp。
*   **SFT 仍是“避风港”**：在固定预算下，智能体更倾向于构建监督学习流水线。在线 RL 仅在 ALFWorld 这种必须交互的任务中展现出作为“最终最佳路径”的优势。
*   **对齐差距的闭合依赖全堆栈**：从非对齐的基础模型（Base）出发，只有最强的智能体堆栈（Claude Code + Opus 4.6）能将其提升至接近指令微调（Instruct）模型的水平。

### 7. 优点
*   **填补空白**：首次系统性评估了智能体“自动化 RL 工程”这一前沿且极具挑战的能力。
*   **诊断性强**：不仅给出一个分数，还通过运行时检测工具揭示了智能体在 RL 闭环中的具体失败模式（如奖励函数设计错误、OOM 崩溃等）。
*   **结构化设计**：三级任务递进设计清晰地界定了“静态自动化”与“动态 RL 工程”的界限。

### 8. 不足与局限
*   **评估噪声**：部分交互式任务（如 DeepSearchQA）的评估本身具有随机性，可能掩盖微小的改进。
*   **非完全因子设计**：由于成本极高，并未对所有“框架 x 驱动模型”组合进行穷尽测试。
*   **策略偏见**：目前的智能体在面对复杂 RL 任务时，往往会退化为复杂的 SFT 组合策略，而非纯粹的在线策略梯度优化，这反映了当前智能体在处理高动态系统时的局限。
