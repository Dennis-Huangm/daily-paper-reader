## 摘要
现有的合成工具使用语料库主要针对离线监督微调（SFT）设计，但强化学习（RL）需要支持奖励可检查在线回放（online rollouts）的可执行环境。我们提出了 COVERT，这是一个两阶段流水线：首先通过带有三级验证的自进化合成生成可靠的基础工具使用轨迹，然后应用保持真值（oracle-preserving）的增强手段，系统性地增加环境复杂度。这些增强引入了干扰工具、间接或模糊的用户查询，以及噪声、多格式或错误的工具输出，同时严格保留原始的工具调用和最终答案作为地面真值（ground truth）。这种设计使得标准案例可以通过参考匹配进行自动奖励计算，而对于错误检测等特殊行为，则可以通过轻量级的评判员辅助验证，从而支持工具调用策略的 RL 优化。在 Qwen2.5-Instruct-14B 上，COVERT-RL 将 BFCL v3 的总准确率从 56.5 提升至 59.9，在 ACEBench 上从 53.0 提升至 59.3，且在通用能力基准测试上的性能退化极小；当在 SFT 之上叠加时，准确率进一步达到 62.1 和 61.8，证实了增量收益。这些结果表明，保持真值的合成环境提供了一个实用的 RL 精炼阶段，与 SFT 互补，能够提高在模糊性和不可靠工具反馈下的工具使用鲁棒性。

## Abstract
Existing synthetic tool-use corpora are primarily designed for offline supervised fine-tuning, yet reinforcement learning (RL) requires executable environments that support reward-checkable online rollouts. We propose COVERT, a two-stage pipeline that first generates reliable base tool-use trajectories through self-evolving synthesis with multi-level validation, and then applies oracle-preserving augmentations that systematically increase environmental complexity. These augmentations introduce distractor tools, indirect or ambiguous user queries, and noisy, multi-format, or erroneous tool outputs, while strictly preserving oracle tool calls and final answers as ground truth. This design enables automatic reward computation via reference matching for standard cases and lightweight judge-assisted verification for special behaviors such as error detection, supporting RL optimization of tool-calling policies. On Qwen2.5-Instruct-14B, COVERT-RL improves overall accuracy on BFCL v3 from 56.5 to 59.9 and on ACEBench from 53.0 to 59.3, with minimal regressions on general-ability benchmarks; when stacked on SFT, it further reaches 62.1 and 61.8, confirming additive gains. These results suggest that oracle-preserving synthetic environments offer a practical RL refinement stage, complementary to SFT, for improving tool-use robustness under ambiguity and unreliable tool feedback.

#### 1. 核心问题与整体含义
*   **研究动机**：现有的工具使用（Tool-use）数据集大多是为监督微调（SFT）设计的静态语料，缺乏强化学习（RL）所需的“在线回放”和“可验证奖励”环境。
*   **核心挑战**：真实世界的工具调用日志稀缺且涉及隐私，而现有的合成数据难以覆盖复杂的异常情况（如干扰工具、模糊查询、工具报错等）。
*   **研究意义**：本文提出了 **COVERT** 框架，旨在构建一个可控、可验证的合成环境，通过强化学习提升大语言模型（LLM）在面对复杂、不可靠工具反馈时的鲁棒性。

#### 2. 方法论
COVERT 采用两阶段流水线构建 RL 训练环境：
*   **第一阶段：可靠基础轨迹生成**
    *   **自进化合成**：从少量种子数据开始，利用 LLM 迭代生成新的工具使用案例。
    *   **多级验证**：通过**格式检查器**（JSON 结构）、**规则检查器**（参数约束、枚举值）和**语义检查器**（LLM 评判逻辑一致性）确保基础数据的绝对可靠。
*   **第二阶段：保持真值的环境增强（Oracle-preserving Augmentation）**
    *   **核心思想**：在修改环境组件（如增加干扰工具、改写查询为间接表达、注入噪声或错误输出）的同时，严格保留原始的“正确工具调用”和“最终答案”作为真值。
    *   **增强类型**：包括干扰工具（Distractors）、查询改写（Ambiguity）、多格式解析、噪声输出、工具报错（Erroneous outputs）等。
*   **奖励设计与 RL 训练**：
    *   **奖励函数**：$R = \lambda_{fmt} R_{format} + \lambda_{cor} R_{correctness}$。
    *   **验证机制**：标准案例采用确定性的参考匹配（Reference Matching）；特殊行为（如检测到工具报错后的反应）采用轻量级 LLM 评判员辅助。
    *   **算法**：采用 **GRPO** 算法进行策略优化，无需额外的价值模型。

#### 3. 实验设计
*   **基座模型**：Qwen2.5-Instruct-7B 和 14B。
*   **对比方法**：
    1.  原始基座模型（Base）。
    2.  TOUCAN-SFT（在 150 万条真实 MCP 数据上微调）。
    3.  COVERT-RL（直接在合成环境上进行 RL）。
    4.  TOUCAN-SFT + COVERT-RL（SFT 后再进行 RL 精炼）。
*   **评测基准**：
    *   **工具调用**：BFCL v3（伯克利函数调用排行榜）、ACEBench。
    *   **通用能力**：LiveCodeBench（代码）、AIME/Math500（数学）、MMLU-Pro（知识）、IFEval（指令遵循）等。
*   **评测协议**：使用统一的“极简提示词”（Minimal Prompt），不提供额外的行为指导，以测试模型的原生鲁棒性。

#### 4. 资源与算力
*   **硬件**：使用 **8 × NVIDIA H200 GPU**。
*   **时长**：7B 模型单次 RL 训练约 **28 小时**；14B 模型约为 7B 的 **1.6 倍**（约 45 小时）。
*   **框架**：基于开源 RL 框架 **VERL**。

#### 5. 实验数量与充分性
*   **实验规模**：涵盖了两个参数量级（7B/14B）的模型，并在工具调用和通用能力两大类共 10 多个 Benchmark 上进行了测试。
*   **充分性**：实验设计了 SFT 与 RL 的叠加对比，验证了两者具有互补性。
*   **客观性**：通过在 held-out（预留）的合成工具集上训练并在完全独立的第三方基准（BFCL, ACEBench）上测试，证明了能力的泛化性而非过拟合。

#### 6 主要结论与发现
*   **性能提升**：COVERT-RL 显著提升了模型在处理“无关查询检测”和“异常输出处理”方面的能力。在 14B 模型上，BFCL v3 提升了 3.4%，ACEBench 提升了 6.3%。
*   **互补性**：RL 与 SFT 结合效果最好（14B 达到 62.1%），说明 SFT 提供结构化先验，而 RL 强化语义正确性和鲁棒性。
*   **通用能力保持**：与 SFT 往往导致通用能力下降不同，COVERT-RL 在提升工具能力的同时，几乎没有在数学、代码和知识类任务上产生退化。

#### 7. 优点
*   **可控性强**：通过增强算子（Knobs）可以精确控制训练环境的难度和扰动类型。
*   **可验证性高**：创新的“保持真值”设计解决了工具调用场景下奖励难以自动计算的痛点。
*   **低成本验证**：通过保留真值元数据，降低了对强评判员（如 GPT-4）的依赖，使用 7B 模型即可完成奖励标注。

#### 8. 不足与局限
*   **长程规划不足**：目前的合成环境主要针对单步或短程多步交互，对于极长路径的复杂智能体规划（Long-horizon planning）模拟尚不充分。
*   **评判员偏差**：虽然引入了真值辅助，但对于“非确定性验证”部分，仍存在 LLM 评判员的主观偏差风险，且文中未进行大规模的人类一致性研究。
*   **环境状态性**：目前的 Mock 环境相对静态，缺乏对复杂状态机（Stateful environments）的深度模拟。
