## 摘要
强化学习 (RL) 已广泛用于训练多轮交互任务中的大语言模型 (LLM) 智能体，但其样本效率受到稀疏奖励和长时序的严重限制。同策略自蒸馏 (OPSD) 通过提供来自拥有地面真值 (ground-truth) 答案的特权教师的密集 Token 级监督来缓解这一问题。然而，这种固定的特权信息无法捕捉智能体任务中多样化的有效策略，且将 OPSD 与 RL 简单结合往往会导致训练崩溃。为了解决这些局限性，我们提出了 Skill-SD，这是一个将智能体自身的轨迹转化为动态的、仅限训练的监督信息的框架。完成的轨迹被总结为简洁的自然语言技能，描述了成功的行为、错误和工作流。这些技能作为动态特权信息仅对教师进行条件化，而学生始终在普通任务提示下行动，并通过蒸馏学习内化这些指导。为了稳定训练，我们推导了一种重要性加权的逆 KL 损失 (reverse-KL loss)，以提供梯度正确的 Token 级蒸馏，并动态地将教师与不断进步的学生同步。在智能体基准测试上的实验结果表明，Skill-SD 显著优于标准 RL 基准，在 AppWorld/Sokoban 上分别提升了原生 GRPO (+14.0%/+10.9%) 和原生 OPD (+42.1%/+40.6%)。项目主页：https://k1xe.github.io/skill-sd/

## Abstract
Reinforcement learning (RL) has been widely used to train LLM agents for multi-turn interactive tasks, but its sample efficiency is severely limited by sparse rewards and long horizons. On-policy self-distillation (OPSD) alleviates this by providing dense token-level supervision from a privileged teacher that has access to ground-truth answers. However, such fixed privileged information cannot capture the diverse valid strategies in agent tasks, and naively combining OPSD with RL often leads to training collapse. To address these limitations, we introduce Skill-SD, a framework that turns the agent's own trajectories into dynamic training-only supervision. Completed trajectories are summarized into compact natural language skills that describe successful behaviors, mistakes, and workflows. These skills serve as dynamic privileged information conditioning only the teacher, while the student always acts under the plain task prompt and learns to internalize the guidance through distillation. To stabilize the training, we derive an importance-weighted reverse-KL loss to provide gradient-correct token-level distillation, and dynamically synchronize the teacher with the improving student. Experimental results on agentic benchmarks demonstrate that Skill-SD substantially outperforms the standard RL baseline, improving both vanilla GRPO (+14.0%/+10.9% on AppWorld/Sokoban) and vanilla OPD (+42.1%/+40.6%). ### 1. 核心问题与研究动机
在多轮交互任务中训练大语言模型（LLM）智能体时，强化学习（RL）面临两大挑战：
*   **奖励稀疏与长路径问题**：任务成功往往只在最后给出一个二元奖励，导致模型难以确定哪些中间步骤或 Token 是有效的。
*   **现有自蒸馏的局限性**：传统的同策略自蒸馏（OPSD）依赖于固定的“地面真值”作为特权信息，但这不适用于路径多样化的智能体任务（如同一个任务有多种 API 调用组合）。此外，直接将自蒸馏与 RL 结合容易导致训练崩溃。

### 2. 核心方法论
Skill-SD 提出了一种将智能体自身轨迹转化为动态、可内化技能的框架：
*   **技能条件化教师（Skill-Conditioned Teacher）**：将智能体过去的轨迹总结为包含“成功分析”、“错误分析”和“黄金工作流”的自然语言**技能**。在训练中，教师模型可以访问这些技能（特权信息），而学生模型仅看到原始任务提示。
*   **重要性加权的逆 KL 损失（Importance-Weighted Reverse-KL Loss）**：为了解决教师和学生因提示词不同导致的分布偏移，论文推导了一种新的损失函数。通过引入重要性采样权重 $\rho$，修正了 Token 级别的梯度偏差，确保蒸馏过程的稳定性。
*   **动态教师同步**：教师模型不再是固定的，而是定期从最新的学生模型检查点同步参数，使特权信号随学生能力的提升而共同进化。
*   **异步技能生成与检索**：使用辅助 LLM 总结轨迹，并利用 UCB（置信上限）算法在技能库中检索最有效的技能，通过异步架构减少训练耗时。

### 3. 实验设计
*   **数据集/场景**：
    *   **AppWorld**：测试 API 协调、多应用状态管理和环境反馈适应能力。
    *   **Sokoban（推箱子）**：测试空间推理和长时序规划能力，具有高度不可逆性。
*   **基准模型**：使用 Qwen3-4B-Instruct-2507 作为基础模型。
*   **对比方法**：
    *   **Vanilla GRPO**：标准群组相对策略优化。
    *   **Skill-Augmented GRPO**：直接在学生提示词中加入技能（不使用蒸馏）。
    *   **Vanilla OPD**：仅使用自蒸馏，不使用 RL 奖励。

### 4. 资源与算力
*   **硬件环境**：使用了 **8 张 NVIDIA L40S GPU**。
*   **软件框架**：基于 `rLLM` 框架和 `verl` 后端实现。
*   **训练细节**：采用了 Ulysses 序列并行（Size 4），学习率为 $1 \times 10^{-6}$，蒸馏系数 $\lambda$ 设为 0.001。

### 5. 实验数量与充分性
*   **实验规模**：在 AppWorld（90 个训练任务，57 个开发任务）和 Sokoban（96 个训练关卡，64 个测试关卡）上进行了完整测试。
*   **消融实验**：非常充分。论文对比了“学生拥有轨迹 vs 教师拥有轨迹”、“动态教师 vs 冻结教师”、“有无重要性权重”等多种配置。
*   **客观性**：通过可视化 Token 概率分布和损失曲线，直观展示了蒸馏如何帮助学生模型对齐教师模型的分布，实验设计严谨。

### 6. 主要结论与发现
*   **性能大幅提升**：Skill-SD 在 AppWorld 上达到 64.9% 的准确率（比 GRPO 高 14.0%），在 Sokoban 上达到 62.5%（比 GRPO 高 10.9%）。
*   **技能应引导教师而非学生**：直接给学生喂技能（Skill-Augmented）会导致严重的过拟合和推理能力下降，而通过蒸馏让学生“内化”技能效果最好。
*   **稳定性关键**：学生拥有的同策略轨迹对稳定训练至关重要；离策略（教师生成轨迹）会导致训练在中期崩溃，尤其是在不可逆任务（如推箱子）中。

### 7. 优点与亮点
*   **理论严密**：针对跨提示词自蒸馏推导了梯度正确的损失函数，解决了 RL 与蒸馏结合时的不稳定性问题。
*   **动态进化**：通过动态同步教师和 UCB 技能检索，构建了一个自我进化的闭环系统。
*   **推理无开销**：学生模型在推理时不需要检索技能或增加提示词长度，保持了轻量化。

### 8. 不足与局限
*   **任务池规模**：实验主要集中在相对较小的任务池上（约 100 个任务），在大规模通用任务上的泛化性有待进一步验证。
*   **检索机制简单**：目前使用 UCB 检索，尚未引入基于向量嵌入（Embedding）的语义检索，可能限制了海量技能库下的表现。
*   **Token 采样限制**：采用 Token 级采样蒸馏而非全词表蒸馏，虽然节省了算力，但可能损失了部分分布信息。
