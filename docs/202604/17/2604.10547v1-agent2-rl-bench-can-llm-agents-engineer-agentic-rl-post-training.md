## 摘要
我们推出了 Agent^2 RL-Bench，这是一个用于评估智能体强化学习（RL）后训练的基准测试，旨在考察 LLM 智能体能否自主设计、实现并运行完整的 RL 流水线，从而提升基础模型。这一能力至关重要，因为 RL 后训练正日益推动模型的对齐和专业化，然而现有的基准测试在很大程度上仍是静态的：仅靠监督微调（SFT）就能取得显著成果，导致交互式 RL 工程化能力未得到测试。Agent^2 RL-Bench 通过三个层级的六个任务解决了这一问题——从静态的基于规则的训练到带有轨迹采集的闭环在线 RL——每个层级都增加了前一层级所不具备的结构性要求。该基准测试提供了带有评分 API 的隔离工作区、记录每次提交和代码修订的运行时检测工具，以及生成结构化运行报告的自动化事后分析，从而实现了对智能体驱动的后训练行为的首次自动化诊断。在涵盖 5 个智能体系统和 6 个驱动 LLM 的多个智能体堆栈中，我们发现智能体取得了显著的交互式增益——在 ALFWorld 上，一个纯 RL 智能体通过 SFT 预热和带有在线 Rollout 的 GRPO，性能从 5.97 提升至 93.28——但在其他任务上仅取得微小进展（DeepSearchQA：+2.75，处于评估噪声范围内）；此外，驱动模型的选择对交互式任务有巨大影响——在相同的脚手架下，更换驱动模型可使交互式改进从接近零变为 +78 个百分点。更广泛地看，该基准测试揭示了在固定预算下，监督流水线在智能体驱动的后训练中占据主导地位，而在线 RL 仅在 ALFWorld 上作为最终的最佳路径取得了成功。代码可在 https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench 获取。

## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. Code is available at https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench.

#### 1. 核心问题与整体含义（研究动机和背景）
论文探讨的核心问题是：**LLM 智能体能否自主完成强化学习（RL）后训练的全流程工程任务？**
随着 DeepSeek-R1 等模型的成功，RL 后训练已成为提升模型推理和对齐能力的关键。然而，现有的智能体基准测试（如 SWE-bench 或 MLE-bench）大多偏向静态代码生成或简单的机器学习流水线，往往通过监督微调（SFT）就能取得高分，无法真正测试智能体在处理**在线数据采集、轨迹奖励分配和闭环 RL 训练**等复杂系统工程时的能力。本研究旨在填补这一空白，评估智能体作为“算法工程师”自主改进基础模型的能力。

#### 2. 方法论：核心思想与技术细节
论文提出了 **Agent^2 RL-Bench**，其核心思想是通过“训练结构覆盖”来评估智能体。
*   **三层级任务结构**：
    *   **L1（静态规则级）**：如 GSM8K，具有确定性验证器，测试基础数据构建。
    *   **L2（静态评判级）**：如 AlpacaEval，引入非确定性的 LLM 评判信号。
    *   **L3（交互式 Rollout 级）**：如 ALFWorld、WebShop，要求智能体实现环境交互、轨迹采集和闭环在线 RL。
*   **关键技术组件**：
    *   **隔离工作区**：为智能体提供只读的基础模型和数据，以及可写的代码/输出目录。
    *   **评分 API**：允许智能体在 12 小时预算内多次提交模型进行评估，获取反馈并迭代。
    *   **运行时仪表化（Instrumentation）**：实时记录智能体的每一次代码修改、提交记录和模型工件。
    *   **自动化事后分析**：自动生成结构化的运行报告，诊断智能体是死于代码错误、奖励设计失败还是训练不稳定。

#### 3. 实验设计
*   **数据集/场景**：涵盖 6 个任务：GSM8K、HumanEval（L1）；AlpacaEval 2.0（L2）；ALFWorld、WebShop、DeepSearchQA（L3）。
*   **智能体系统（Scaffolds）**：对比了 OpenHands、OpenCode、Claude Code、Codex CLI 和 Gemini CLI。
*   **驱动模型（Driver LLMs）**：使用了 GPT-5.4、GPT-5.2、GPT-4o、Claude 3.5/4.5/4.6 系列（注：文中使用了前瞻性命名如 GPT-5.4/Opus 4.6）以及 Gemini 2.5-flash。
*   **目标模型（Base Models）**：主要针对 Qwen2.5-7B-Instruct 和未对齐的 Qwen3-8B-Base 进行后训练改进。

#### 4. 资源与算力
*   **时间预算**：每个任务为智能体分配 **12 小时** 的自主工程时间。
*   **算力设备**：附录中提到实验使用了 **H200 GPU**（如在 DeepSearchQA 追踪实验中使用了 2×H200）。
*   **训练细节**：智能体自主决定训练参数（如 LoRA rank、学习率、Epochs 等）。

#### 5. 实验数量与充分性
*   **实验规模**：进行了两组主要研究：一是 7B 指令微调模型的脚手架对比；二是 8B 基础模型的受控变量研究（固定脚手架换驱动模型）。
*   **充分性与公平性**：
    *   **优点**：引入了 `pass@k` 分析和种子扫描（20 次重复）来评估 DeepSearchQA 的评估噪声，确保结论客观。
    *   **局限**：由于成本极高，部分 8B 实验为单次运行（Single-run），缺乏完整的置信区间统计。实验设计并非完全的析因实验（Factorial Design），即未测试所有可能的脚手架-驱动模型组合。

#### 6. 主要结论与发现
*   **交互式增益显著但非普遍**：在 ALFWorld 上，智能体能将模型得分从 5.97 提升至 93.28；但在 DeepSearchQA 上几乎没有实质性进展（仅 +2.75，处于噪声区间）。
*   **驱动模型是决定性因素**：在相同脚手架下，更换驱动模型可导致 ALFWorld 的改进幅度产生 **78 个百分点** 的巨大差异。
*   **SFT 仍是“避风港”**：在固定预算下，智能体更倾向于构建复杂的 SFT 流水线（如轨迹过滤后微调），而非不稳定的在线 RL。在线 RL 仅在 ALFWorld 任务中成为最优路径。
*   **对齐差距的闭合依赖全堆栈**：对于未对齐的基础模型，只有最强的智能体堆栈（如 Claude Code + 高阶驱动）能将其提升至指令微调模型的水平，弱堆栈甚至会导致模型性能倒退。

#### 7. 优点（亮点）
*   **结构化诊断能力**：不仅看最终分数，还能通过自动化报告分析智能体在哪个环节（如 Rollout 构建、奖励设计）失败。
*   **填补基准空白**：首次系统性地测试了智能体在“在线交互式 RL”这一高难度工程领域的表现。
*   **真实的迭代过程**：模拟了人类工程师“编写-运行-诊断-修改”的循环，而非单次代码生成。

#### 8. 不足与局限
*   **评估噪声**：DeepSearchQA 等任务存在较高的评估方差（±3pp），可能误导智能体的迭代方向。
*   **计算成本高昂**：12 小时的运行限制了大规模重复实验的可能性。
*   **任务覆盖面**：虽然涵盖了三个层级，但任务总数（6 个）相对较少，可能存在特定任务的偏见。
*   **策略偏好**：实验观察到智能体在 ALFWorld 的成功部分依赖于 SFT 组合策略，而非纯粹的算法创新。
