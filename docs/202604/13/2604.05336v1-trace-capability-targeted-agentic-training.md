## 摘要
部署在智能体环境中的大语言模型（LLMs）必须在不同的任务实例中运用多种能力，其中能力是指在轨迹中执行一个或多个对于成功解决环境中特定子集任务所必需的动作。许多现有方法要么依赖于并非针对模型在目标环境中实际能力缺陷的合成训练数据，要么直接在目标环境上进行训练，从而要求模型在不同任务间隐式地学习这些能力。我们引入了 TRACE（将循环出现的智能体失败转化为针对能力的训练环境），这是一个用于特定环境下智能体自我提升的端到端系统。TRACE 通过对比成功和失败的轨迹来自动识别缺失的能力，为每种能力合成一个针对性的训练环境，并根据是否运用了该能力提供奖励，随后在每个合成环境上通过强化学习（RL）训练 LoRA 适配器，并在推理时路由到相关的适配器。实验表明，TRACE 在不同环境中具有良好的泛化性，在 τ²-bench（客户服务）上比基础智能体提升了 14.1 分，在 ToolSandbox（工具使用）上增加了 7 个满分，分别比最强的基准模型高出 7.4 分和 4 个满分。在给定相同采样次数的情况下，TRACE 的扩展效率高于基准模型，在 τ²-bench 上分别超过 GRPO 和 GEPA 9.2 分和 7.4 分。

## Abstract
Large Language Models (LLMs) deployed in agentic environments must exercise multiple capabilities across different task instances, where a capability is performing one or more actions in a trajectory that are necessary for successfully solving a subset of tasks in the environment. Many existing approaches either rely on synthetic training data that is not targeted to the model's actual capability deficits in the target environment or train directly on the target environment, where the model needs to implicitly learn the capabilities across tasks. We introduce TRACE (Turning Recurrent Agent failures into Capability-targeted training Environments), an end-to-end system for environment-specific agent self-improvement. TRACE contrasts successful and failed trajectories to automatically identify lacking capabilities, synthesizes a targeted training environment for each that rewards whether the capability was exercised, and trains a LoRA adapter via RL on each synthetic environment, routing to the relevant adapter at inference. Empirically, TRACE generalizes across different environments, improving over the base agent by +14.1 points on $τ^2$-bench (customer service) and +7 perfect scores on ToolSandbox (tool use), outperforming the strongest baseline by +7.4 points and +4 perfect scores, respectively. Given the same number of rollouts, TRACE scales more efficiently than baselines, outperforming GRPO and GEPA by +9.2 and +7.4 points on $τ^2$-bench.

*   **核心问题**：大型语言模型（LLM）智能体在复杂环境中往往因为缺乏某些特定能力（如结构化数据推理、工具调用精度等）而失败。
*   **研究动机**：
    *   **现有方法局限**：直接在目标环境进行强化学习（RL）会导致奖励信号稀疏，模型难以将失败归因于具体的能力缺失；而使用通用的合成数据训练则缺乏针对性，无法解决模型在特定环境下的“短板”。
    *   **信用分配难题**：在多步交互中，最终的成败难以直接指导中间环节某种特定能力的改进。
*   **目标**：建立一个自动化的系统，能够识别模型“不会什么”，并针对性地“教它什么”。

### 2. 方法论：TRACE 系统流程
TRACE（Turning Recurrent Agent failures into Capability-targeted training Environments）包含四个关键阶段：
1.  **对比能力识别（Contrastive Identification）**：
    *   让基础智能体在目标环境中运行，收集成功和失败的轨迹。
    *   利用 LLM 分析智能体对比两类轨迹，识别出导致失败的候选能力，并计算“对比差距”（即该能力在失败组中缺失但在成功组中存在的频率差）。
2.  **环境合成（Environment Synthesis）**：
    *   针对识别出的每种能力缺陷，自动生成一个专门的合成训练环境。
    *   该环境隔离特定能力，确保任务成功高度依赖于该能力的运用，并提供自动化的程序化验证（基于状态变化或工具参数）。
3.  **针对性强化学习（Targeted RL）**：
    *   在每个合成环境上，使用 **GRPO（组相对策略优化）** 算法训练独立的 **LoRA 适配器**。
    *   GRPO 通过组内奖励归一化，使训练信号对不同环境的奖励尺度具有不变性。
4.  **动态路由推理（Routing）**：
    *   推理时，由基础模型根据当前任务描述，从候选能力库中选择最匹配的一个适配器进行激活。

### 3. 实验设计
*   **数据集/场景**：
    *   **$\tau^2$-Bench**：模拟真实世界的客户服务场景（航空和零售领域），涉及复杂的策略约束和多步工具交互。
    *   **ToolSandBox**：一个状态化的工具使用基准测试，侧重于权限错误恢复和时间推理等能力。
*   **对比方法**：
    *   **Base Model**：未经针对性训练的基础模型（Qwen3-30B）。
    *   **直接 RL (GRPO on Target)**：直接在目标基准测试任务上进行 RL 训练。
    *   **通用合成数据方法**：如 AWM（智能体世界模型）和 ADP（智能体数据协议）。
    *   **提示词优化 (GEPA)**：通过进化算法优化提示词中的能力描述。
    *   **模型合并方法**：如 CORE-TSV 等尝试将多个适配器合并为一个的技术。

### 4. 资源与算力

### 5. 实验数量与充分性
*   **实验规模**：在两个不同的复杂 Benchmark 上进行了验证，涵盖了从识别稳定性到扩展性分析的多个维度。
*   **充分性**：
    *   **稳定性测试**：通过 10 次独立运行验证了能力识别过程的收敛性。
    *   **消融实验**：对比了单能力训练、多能力混合训练、模型合并以及动态路由的效果。
    *   **扩展性实验**：展示了性能随采样次数（Rollouts）增加的增长曲线，证明了 TRACE 比基线方法更具采样效率。
*   **客观性**：在相同的采样预算下对比了多种强基线，实验设计较为公平。

### 6. 主要结论与发现
*   **性能显著提升**：在 $\tau^2$-Bench 上比基础模型提升了 **14.1%** 的成功率，在 ToolSandBox 上获得了更多的满分。
*   **扩展效率高**：在相同的训练采样量下，TRACE 的表现优于直接在目标环境训练的 GRPO 和提示词优化方法 GEPA。
*   **路由优于合并**：实验发现，动态路由到特定能力的适配器，比将所有能力强行训练进一个模型或进行权重合并效果更好，避免了能力间的干扰。
*   **训练优于提示**：即使是经过优化的提示词，其效果也显著低于通过 RL 显式训练出的能力适配器。

### 7. 优点
*   **精准打击**：通过对比分析精准定位模型弱点，避免了盲目的大规模合成数据训练。
*   **模块化**：每个能力是一个独立的 LoRA 适配器，易于扩展和维护，不会导致灾难性遗忘。
*   **自动化闭环**：实现了从失败诊断到环境生成再到模型改进的全流程自动化，减少了人工干预。

### 8. 不足与局限
*   **路由开销**：推理时需要先进行一次路由分类，虽然开销较小，但在极高性能要求的场景下仍是额外负担。
*   **单适配器限制**：目前的路由机制每次只激活一个适配器。如果一个极其复杂的任务需要同时运用多个新训练的能力，当前的单路由方案可能无法达到最优。
*   **依赖分析模型**：系统的高度自动化依赖于“分析智能体”的质量，如果分析模型无法准确识别失败原因，后续生成的环境将失效。
