## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. Code is available at https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench.

### Agent^2 RL-Bench: 评估 LLM 智能体自动化强化学习工程能力的基准测试

#### 1. 论文的核心问题与整体含义
随着强化学习（RL）后训练（Post-training）成为大模型对齐和专业化的核心驱动力，一个关键问题浮出水面：**LLM 智能体能否像人类专家一样，自主设计、实现并运行完整的 RL 流水线来提升基础模型？**
现有的 Agent 基准测试（如 SWE-bench 或 MLE-bench）大多侧重于代码生成或通用的机器学习任务，且往往可以通过静态的监督微调（SFT）取得高分，无法真正测试智能体在处理交互式环境、轨迹收集、奖励函数设计及闭环 RL 训练等复杂工程问题的能力。本论文提出了 **Agent^2 RL-Bench**，旨在填补这一空白，评估“智能体化 RL 后训练（Agentic RL Post-training）”的能力。

#### 2. 论文提出的方法论
论文的核心思想是**训练结构覆盖（Training Structure Coverage）**，将任务按复杂度分为三个等级：
*   **L1：静态规则任务（GSM8K, HumanEval）**。测试基础的数据构建和监督训练，具有确定性的验证器。
*   **L2：静态判别任务（AlpacaEval 2.0）**。引入非确定性的 LLM 裁判信号，要求智能体针对偏好信号进行优化。
*   **L3：交互式 Rollout 任务（ALFWorld, WebShop, DeepSearchQA）**。这是核心挑战，要求智能体与状态环境交互、收集多步轨迹、处理稀疏/稠密奖励，并维持在线数据收集循环。

**关键技术细节：**
*   **隔离工作区与评分 API**：为 Agent 提供只读的基础模型和训练数据，通过 API 提交候选模型并获取反馈，模拟真实的迭代开发过程。
*   **运行时监测（Instrumentation）**：记录 Agent 的每一次代码修改、模型提交和运行日志。
*   **自动化事后分析**：自动生成结构化的运行报告和案例研究，诊断 Agent 是如何搜索策略、为何失败以及如何适配的。

#### 3. 实验设计
*   **数据集/场景**：涵盖数学推理（GSM8K）、代码生成（HumanEval）、指令遵循（AlpacaEval）、具身智能（ALFWorld）、电商搜索（WebShop）和深度搜索问答（DeepSearchQA）。
*   **对比方法（Agent 栈）**：
    *   **5 个 Agent 系统**：OpenHands, OpenCode, Claude Code, Codex CLI, Gemini CLI。
    *   **6 个驱动 LLM**：GPT-5.4, GPT-5.2, GPT-4o, Claude 3.5 Opus/Sonnet, Gemini 2.5-flash。
*   **消融实验**：针对 Claude Code 设置了四种模式：仅 SFT、仅 RL、自由模式（Free）和多任务模式（Multi），以探究不同训练范式的影响。

#### 4. 资源与算力
*   **时间预算**：每个任务为 Agent 分配了固定的 **12 小时**（部分实验为 24 小时）工程预算。
*   **硬件设备**：文中提到在随访实验中使用了 **2 × H200 GPU**（针对 DeepSearchQA 任务）。
*   **算力消耗**：附录中详细列出了 Token 使用量，例如 Claude Code 在 19 个 Agent 运行中消耗了约 2.9 亿个 Token，估计成本约为 796 美元。

#### 5. 实验数量与充分性
*   **实验规模**：论文进行了跨 5 个系统、6 个驱动模型和 6 个任务的大规模测试。
*   **充分性**：实验设计较为全面，不仅有最终得分，还包含了 Scaling 分析（时间、Token、提交次数三个维度）和 Pass@k 分析。
*   **客观性与公平性**：通过隔离测试集防止泄漏，并使用统一的 12 小时预算。但作者也指出，8B 模型的受控研究并非完全的因子实验（未测试所有组合），且部分任务（如 DeepSearchQA）存在较高的评估噪声。

#### 6. 论文的主要结论与发现
*   **交互式增益显著但极不均衡**：在 ALFWorld 上，RL 模式 Agent 将性能从 5.97 提升至 93.28；但在 DeepSearchQA 上，所有 Agent 几乎都无法突破基准（仅提升约 2.75，处于噪声范围内）。
*   **驱动模型（Driver）至关重要**：在相同框架下，更换驱动模型可使 ALFWorld 的提升从接近 0 变为 +78pp。
*   **SFT 仍是主导策略**：在固定预算下，Agent 倾向于回退到 SFT 流水线，仅在 ALFWorld 中在线 RL 表现出绝对优势。
*   **对齐差距的弥补依赖全栈能力**：从非对齐的基础模型（Qwen3-8B-Base）出发，只有最强的 Agent 栈（Claude Code + Opus 4.6）能达到或超过指令微调版的水平。

#### 7. 优点
*   **结构化诊断**：通过分级任务清晰地界定了静态自动化与交互式 RL 工程之间的能力边界。
*   **深度行为分析**：不仅关注分数，还通过自动化报告揭示了 Agent 的“心路历程”（如从 SFT 到 DPO 的演进）。
*   **真实工程模拟**：引入了 OOM（显存溢出）处理、超参数搜索和模型融合等真实工程挑战。

#### 8. 不足与局限
*   **任务覆盖范围**：虽然涵盖了 6 类任务，但对于更复杂的 RL 场景（如多智能体竞争、极长程规划）覆盖不足。
*   **评估噪声**：DeepSearchQA 等任务的评估方差较大（±3pp），导致微小提升的可靠性受限。
*   **内容过滤干扰**：实验中提到 Azure 的内容过滤器干扰了约 4% 的 AlpacaEval 裁判调用，可能对结果产生轻微偏差。
*   **计算成本**：该基准测试对算力要求极高（Agent 需要频繁训练模型），限制了其在小规模研究中的普及。
