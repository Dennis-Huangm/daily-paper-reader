## 摘要
使用大语言模型（LLMs）自动化优化建模是实现可扩展决策智能的一条充满前景的路径，但现有方法要么依赖于基于闭源 LLM 构建且推理延迟较高的智能体流水线，要么使用昂贵的过程监督微调较小的 LLM，而这往往会过拟合到单一的求解器 API。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种执行验证的学习框架，将数学规划求解器视为确定性的交互式验证器。给定自然语言问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在“生成-执行-反馈-更新”的闭环过程中进行优化。这种仅基于结果的表述消除了对过程级监督的需求，并通过切换验证环境而非重建求解器特定数据集，实现了跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 上跨 Gurobi、OR-Tools 和 COPT 的实验表明，EVOM 的表现达到或超过了经过过程监督的 SFT，支持零样本求解器迁移，并通过在目标求解器后端下持续训练实现了有效的低成本求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

### 2. 方法论：核心思想与技术细节
EVOM 的核心思想是**将数学规划求解器视为一个确定性的、交互式的验证器**，采用“仅结果监督”（Outcome-only supervision）的强化学习模式。

*   **核心流程（生成-执行-反馈-更新）**：
    1.  **生成**：给定问题描述和目标求解器标识，策略模型生成包含“思考（think）”和“代码（code）”的结构化响应。
    2.  **执行**：在沙盒环境中运行生成的 Python 代码，调用目标求解器（如 Gurobi）。
    3.  **反馈（奖励设计）**：
        *   **格式奖励 ($r_{fmt}$)**：检查输出是否符合指定的 XML 标签格式。
        *   **结果奖励 ($r_{ans}$)**：将求解器输出的目标值（Objective Value）或状态（如不可解）与标准答案进行比对。若数值在误差范围内或状态一致，则给予正奖励。
    4.  **更新**：使用 **GRPO**（群体相对策略优化）或 **DAPO** 算法，通过计算组内相对优势来更新模型参数，无需额外的评论家（Critic）模型。
*   **关键技术**：
    *   **求解器调节（Solver-Conditioning）**：在 Prompt 中加入求解器名称，使模型学习不同 API 的语法。
    *   **环境切换（Environment Switching）**：通过简单更换后端求解器即可实现模型适配，无需重新构建数据集。

### 6. 主要结论与发现
*   **结果监督优于过程监督**：仅利用求解器反馈的 EVOM 在多个基准上达到或超过了使用昂贵过程标注的 SFT 模型。

### 7. 优点（亮点）

*   **闭环自进化**：通过“生成-执行-反馈”的闭环，模型能够从错误中学习，特别是纠正变量类型错误和 API 调用错误。
*   **解耦逻辑与语法**：成功证明了数学建模逻辑可以与特定求解器的编程语法解耦。

### 8. 不足与局限
*   **冷启动依赖**：对于预训练语料中极度罕见的求解器，纯 RL 探索效率低，仍需少量 SFT 数据进行“热身”。
*   **复杂逻辑瓶颈**：错误分析显示，虽然 RL 解决了大部分代码实现和变量定义错误，但在处理极其复杂的约束逻辑（Semantic Errors）时，提升幅度相对有限。
*   **奖励稀疏性**：对于非常困难的问题，如果模型始终无法生成可运行的代码，将无法获得有效的奖励信号。
