## 摘要
利用大语言模型（LLM）实现自动化优化建模是通往可扩展决策智能的一条极具前景的路径，但现有方法要么依赖于基于高推理延迟的闭源 LLM 构建的智能体流水线，要么使用昂贵且往往过度拟合单一求解器 API 的过程监督来微调较小的 LLM。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种将数学规划求解器视为确定性交互式验证器的执行验证学习框架。给定自然语言问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在“生成-执行-反馈-更新”的闭环过程中进行优化。这种仅基于结果的表述消除了对过程级监督的需求，并通过切换验证环境而非重构特定于求解器的数据集来实现跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 上针对 Gurobi、OR-Tools 和 COPT 进行的实验表明，EVOM 的性能达到或超过了经过过程监督微调（SFT）的模型，支持零样本求解器迁移，并通过在目标求解器后端下持续训练实现了有效的低成本求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

### 1. 核心问题与整体含义
*   **研究动机**：将自然语言描述的实际问题转化为数学优化模型（如线性规划、整数规划）是一项高度专业且复杂的任务。
*   **现有挑战**：
    *   **Agent 方案**：依赖闭源大模型（如 GPT-4），推理延迟高且成本昂贵。
    *   **SFT 方案**：依赖高质量的“过程监督”（即中间推理步骤和参考代码的标注），标注成本极高，且模型容易过度拟合特定求解器（如 Gurobi）的 API 语法，缺乏跨求解器的泛化能力。
*   **核心目标**：探索是否可以像数学竞赛（Math RL）一样，利用求解器作为“客观验证器”，通过强化学习（RL）在没有过程标注的情况下，仅靠最终结果反馈来提升模型的优化建模能力。

### 2. 方法论：EVOM 框架
*   **核心思想**：提出 **EVOM (Execution-Verified Optimization Modeling)**，将数学规划求解器（Solver）视为一个确定性的、交互式的验证环境。
*   **关键流程**：
    1.  **生成 (Generate)**：模型接收“问题 + 目标求解器标识”，生成包含 `<think>`（推理过程）和 `<code>`（求解代码）的结构化响应。
    2.  **执行 (Execute)**：在沙箱环境中运行生成的代码，调用目标求解器（如 Gurobi, OR-Tools, COPT）。
    3.  **反馈 (Feedback)**：捕获执行状态（是否报错）、求解状态（是否最优）及目标函数值（Objective Value）。
    4.  **更新 (Update)**：将执行结果转化为标量奖励，利用 **GRPO**（组相对策略优化）或 **DAPO** 算法进行策略更新。
*   **奖励函数设计**：
    *   **格式奖励 ($r_{fmt}$)**：确保模型遵循特定的 XML 标签格式。
    *   **结果奖励 ($r_{ans}$)**：仅根据求解器输出的数值与标准答案的接近程度（容差范围内）给予奖励，不参考任何中间代码实现。

### 6. 主要结论与发现
*   **结果胜过过程**：仅使用结果监督的 EVOM 性能达到或超过了使用昂贵过程监督的 SFT 模型（如 ORLM）。
*   **强大的泛化性**：EVOM 训练的模型在跨求解器迁移时表现稳健，而 SFT 模型因过度拟合特定 API 语法在切换求解器时性能会发生崩溃。
*   **思考的重要性**：强化学习诱导出的“思考过程”不仅是解释，更是模型内部的逻辑工作区，对处理复杂约束至关重要。
*   **小模型潜力**：经过 EVOM 训练的 3B 模型在多个指标上接近或超过了 7B 基座模型。

### 7. 优点与亮点
*   **低成本数据方案**：摆脱了对“推理链”标注的依赖，仅需“问题-答案对”即可训练。
*   **环境解耦**：通过将求解器视为环境的一部分，实现了“环境切换即适配”，极大降低了迁移到新求解器的成本。
*   **闭环验证**：利用求解器的确定性反馈消除了奖励模型（Reward Model）可能带来的偏差（Reward Hacking）。

### 8. 不足与局限
*   **复杂逻辑瓶颈**：虽然在变量类型和 API 调用上进步明显，但在处理极其复杂的语义约束（需要深层逻辑推理）时，提升幅度相对有限。
*   **冷启动依赖**：对于预训练阶段完全未见过的求解器，仍需要极少量（约 100 条）的 SFT 数据来引导模型掌握基本的 API 语法，无法完全从零开始。
*   **应用限制**：目前主要集中在数学规划类问题，对于非结构化或缺乏确定性求解器的决策问题尚待验证。
