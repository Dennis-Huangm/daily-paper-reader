## 摘要
我们推出了 Agent^2 RL-Bench，这是一个用于评估智能体强化学习（agentic RL）后训练的基准测试，旨在考察大语言模型（LLM）智能体能否自主设计、实现并运行完整的强化学习流水线，从而提升基础模型的能力。这一能力至关重要，因为强化学习后训练正日益成为模型对齐和专业化的驱动力，然而现有的基准测试大多仍是静态的：仅靠监督微调（SFT）就能取得显著成果，导致交互式强化学习工程化能力未得到测试。Agent^2 RL-Bench 通过三个等级的六个任务解决了这一问题——从静态的基于规则的训练到带有轨迹采集的闭环在线强化学习——每个等级都增加了前一等级所不具备的结构性要求。该基准测试提供了带有评分 API 的隔离工作区、记录每次提交和代码修订的运行时检测工具，以及生成结构化运行报告的自动化事后分析，从而实现了对智能体驱动的后训练行为的首次自动化诊断。在涵盖 5 个智能体系统和 6 个驱动 LLM 的多个智能体堆栈中，我们发现智能体取得了显著的交互式增益——在 ALFWorld 上，一个纯强化学习智能体通过 SFT 预热和带有在线展开（online rollouts）的 GRPO，其性能从 5.97 提升至 93.28——但在其他任务上仅取得微小进展（如 DeepSearchQA 仅提升 2.75，处于评估噪声范围内）；此外，驱动模型的选择对交互式任务有巨大影响——在相同的框架下，更换驱动模型可使交互式改进从接近零变为提升 78 个百分点。更广泛地说，该基准测试揭示了在固定预算下，监督流水线在智能体驱动的后训练中占据主导地位，而在线强化学习仅在 ALFWorld 上作为最终的最佳路径取得了成功。

## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. Code is available at https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench.

### 论文总结：Agent^2 RL-Bench —— LLM 智能体能否工程化实现强化学习后训练？

#### 1. 论文的核心问题与整体含义（研究动机和背景）
随着强化学习（RL）在模型对齐和推理能力提升（如 DeepSeek-R1, InstructGPT）中展现出核心作用，构建 RL 流水线已成为模型开发的关键。然而，这通常需要专家在奖励设计、算法选择、稳定性控制和在线数据采集方面进行复杂决策。
论文提出了**智能体强化学习后训练（Agentic RL Post-training）**的概念：即 LLM 智能体能否在有限预算内，自主设计、实现、调试并运行一个完整的后训练流水线来提升基础模型。现有的基准测试（如 SWE-bench 或 MLE-bench）大多偏向静态代码生成或简单的 ML 任务，往往通过监督微调（SFT）就能取得好成绩，无法有效测试智能体处理**在线交互、轨迹采集和长程奖励优化**的能力。

#### 2. 论文提出的方法论
核心思想是构建一个具有**结构化覆盖（Training Structure Coverage）**的基准测试，将任务按难度和结构分为三个层级：
*   **L1：静态规则任务（GSM8K, HumanEval）**。测试基础的数据构建和监督训练能力，奖励由确定性校验器给出。
*   **L2：静态评判任务（AlpacaEval 2.0）**。引入非确定性的 LLM 评判信号，测试智能体在非交互环境下的偏好优化能力。
*   **L3：交互式展开任务（ALFWorld, WebShop, DeepSearchQA）**。这是核心挑战，要求智能体与环境交互、采集多步轨迹、处理稀疏或稠密奖励，并维持在线训练闭环。

**关键技术细节：**
*   **隔离工作区与评分 API**：为智能体提供独立环境，通过 API 进行多次迭代提交，模拟真实的工程诊断过程。
*   **运行时仪表化（Runtime Instrumentation）**：记录智能体的每一次代码修改、模型提交和运行日志。
*   **自动化事后分析**：利用 LLM 自动生成结构化的运行报告，诊断智能体是死于代码错误、奖励设计失败还是训练不稳定。

#### 3. 实验设计
*   **数据集/场景**：涵盖数学推理（GSM8K）、代码生成（HumanEval）、指令遵循（AlpacaEval）、具身智能（ALFWorld）、网络购物交互（WebShop）和深度搜索问答（DeepSearchQA）。
*   **基准模型（Base Models）**：Qwen2.5-7B-Instruct 和 Qwen3-8B-Base（未对齐版本）。
*   **对比方法（智能体堆栈）**：
    *   **5 种智能体框架（Scaffolds）**：OpenHands, OpenCode, Claude Code, Codex CLI, Gemini CLI。
    *   **6 种驱动模型（Driver LLMs）**：GPT-5.4, Claude 4.6 Opus, Claude 4.5 Sonnet, GPT-5.2, GPT-4o, Gemini 2.5-flash。
*   **操作模式消融**：针对 Claude Code 设置了 SFT-only、RL-only、Free（自由选择）和 Multi（多任务）四种模式。

#### 4. 资源与算力
*   **工程预算**：每个任务给予智能体 **12 小时**（部分实验为 24 小时）的自主工程时间。
*   **硬件资源**：实验中提到了使用 **H200 GPU**（例如在 DeepSearchQA 的对照实验中使用 2×H200）。
*   **成本估算**：论文附录详细列出了 Token 消耗，例如 Claude Code 在 19 个智能体运行中消耗了约 2.9 亿个 Token，估计成本约为 796 美元。

#### 5. 实验数量与充分性
*   **实验规模**：涵盖了 5 个系统、6 个驱动模型、2 个基准模型，并在 6 个任务上进行了测试。
*   **消融与对照**：进行了驱动模型与框架的解耦实验（在 Qwen3-8B 上固定框架换驱动），以及操作模式的对比实验。
*   **客观性与公平性**：
    *   采用了单次运行（12h 预算）和多次运行（pass@k）相结合的评估方式。
    *   针对 DeepSearchQA 等高方差任务，进行了 20 次重复实验以确定噪声范围（±3pp）。
    *   **局限性**：由于成本极高，部分 8B 模型的实验为单次运行，缺乏完整的置信区间。

#### 6. 论文的主要结论与发现
*   **交互式增益显著但非通用**：在 ALFWorld 上，智能体能将模型性能从 5.97 提升至 93.28（通过 GRPO 在线强化学习），但在 DeepSearchQA 上几乎没有实质进展。
*   **驱动模型是决定性因素**：在相同框架下，更换驱动模型（如从 GPT-4o 换到 GPT-5.2）在交互任务上的表现差异可达 **78 个百分点**。
*   **SFT 依然占据主导**：在固定预算下，智能体更倾向于构建复杂的 SFT 流水线。在线 RL 仅在 ALFWorld 这一特定任务中成为最优路径。
*   **对齐缺口（Alignment Gap）**：对于未对齐的 Base 模型，强力智能体堆栈能将其提升至 Instruct 级别，而弱智能体甚至会导致性能退化。

#### 7. 优点
*   **填补空白**：首次系统性评估了智能体“工程化 RL”的能力，而非仅仅是“写 RL 代码”的能力。
*   **诊断深度**：通过运行时记录和自动化报告，能够清晰区分智能体是“代码写不出”还是“算法不收敛”。
*   **现实性强**：引入了 12 小时预算和迭代提交机制，非常贴近真实的人类工程师工作流程。

#### 8. 不足与局限
*   **任务多样性有限**：虽然涵盖了三个层级，但总共只有 6 个任务，可能无法完全代表复杂的工业级后训练场景。
*   **评估噪声**：AlpacaEval 受 Azure 内容过滤影响，DeepSearchQA 存在较高的评估方差，影响了微小改进的可信度。
*   **非完全因子设计**：由于算力成本，未能测试所有“框架+驱动模型”的组合，部分结论带有暗示性而非绝对证明。
*   **应用限制**：目前智能体在处理极其复杂的 RL 稳定性问题（如 DeepSearchQA）时依然表现乏力。
